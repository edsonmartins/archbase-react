# Conventions

Esses padrões servem para manter a API consistente (e o catálogo/llms alinhados):

## Naming
- Prefixo `Archbase` para todos os componentes públicos.
- Hooks começam com `use`, providers terminam em `Provider`.
- Helpers e renderers usam `renderX` (`renderText`, `renderCurrency` — veja o catálogo para o link).

## onChange / value / defaultValue
- Prefira `onChangeValue(value, event?)` para componentes data-aware (`ArchbaseEdit`, `ArchbaseMaskEdit`).
- Quando houver `defaultValue`, documente no catálogo (JSDoc `@default`); o storybook `archbase-edit` mostra o comportamento.
- Documente claramente se o component entende `dataSource` vs `value` (ex: `ArchbaseEdit` usa `IArchbaseDataSource`).

## Styling
- Explicite quando expõe `className`, `styles`, `classNames` ou `variant`.
- Use o `component-catalog.json` para verificar quais props de styling existem (`catalog.components.find(c => c.name === 'ArchbaseDataGrid').props`).

## Status e egressos
- Use `@status stable | experimental | legacy` no JSDoc para o catálogo capturar o valor automaticamente.
- Quando um componente estiver em processo de migração, atualize `llms.txt` e `DOCS_AUDIT.md` citando o `canonicalUrl`.

## checklist
- Toda story deve ter `Meta` com `tags: ['autodocs']`.
- Recipes MDX (ex: `docs/stories/FormRecipes.stories.mdx`) devem referenciar os links do catálogo (`component-catalog.json`).
